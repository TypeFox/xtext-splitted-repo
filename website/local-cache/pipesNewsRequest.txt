<p>One of the key technologies of JavaFX is that one uses CSS to theme the whole application but the default CSS-Editors are not of much help because the properties used are specific to JavaFX (read they are all customized extension starting with -fx).</p>
<p>… and because I decided already some time ago that I had to teach myself how to write my own DSL using Xtext it was the prefect chance to create a CSS-Editor for JavaFX using Xtext.</p>
<p>… and because I was traveling a lot in the last 2 weeks (I’ve been to 3 democamps) I had a lot of spare time to work on something I’d like to call “e(fx)clipse” a start of an Eclipse Tooling for JavaFX.</p>
<p>So what do I have for Release 0.0.1 of the Tooling? Well all I have is a start for a CSS-Editor built using Xtext.</p>
<p>Here are 2 screenshots of the current CSS-Editor running in Eclipse 4.1 (it runs as well in 3.7 naturally).</p>
<p><a rel="nofollow" target="_blank" href="http://tomsondev.files.wordpress.com/2011/06/fx-tooling.png"><img alt="" class="alignnone size-full wp-image-2325" height="279" src="http://tomsondev.files.wordpress.com/2011/06/fx-tooling.png?w=588&h=279" title="fx-tooling" width="588"/></a></p>
<p><a rel="nofollow" target="_blank" href="http://tomsondev.files.wordpress.com/2011/06/fx-tooling-1.png"><img alt="" class="alignnone size-full wp-image-2326" height="278" src="http://tomsondev.files.wordpress.com/2011/06/fx-tooling-1.png?w=588&h=278" title="fx-tooling-1" width="588"/></a></p>
<p>So as you see the editor has not too many features as of now:</p>
<ul>
<li>Basic Syntax-Highlighting</li>
<li>Proposals for CSS-Attributes</li>
<li>Support for Attribute value proposals</li>
</ul>
<p>but I hope to improve the initial features.</p>
<p>BTW one of the main features of the CSS-Editor is that it allows people to plug in their own Property-Rules so e.g. e4 could plug-in their own rules, so one can for mozilla, webkit, … -specific rules.</p>
<p>As of now I have 2 property-sets:</p>
<ul>
<li>CSS 2.1</li>
<li>JavaFX</li>
</ul>
<p>but they are not part of the core plugin but contributed simply as OSGi-Services <img alt=":-)" class="wp-smiley" src="http://s0.wp.com/wp-includes/images/smilies/icon_smile.gif"/> </p>
<p>So if you want to give the editor a spin. You can download the p2-repo from my github <a rel="nofollow" target="_blank" href="https://github.com/downloads/tomsontom/e-fx-clipse/at.bestsolution.efxclipse.tooling.updatesite-0.0.1.zip">account</a>. Please note this is a very very early release and the editor currently is only registered for .fxcss-Files.</p>
<p>Update:<br />
There’s one JavaFX syntax not yet support by the editor showing an error marker but the problem is not the CSS-Editor but in reality the CSS-Definition speced by <a rel="nofollow" target="_blank" href="http://download.oracle.com/docs/cd/E17802_01/javafx/javafx/1.3/docs/api/javafx.scene/doc-files/cssref.html">JavaFX</a>. I’ve already filed a <a rel="nofollow" target="_blank" href="http://javafx-jira.kenai.com/browse/RT-14402">Jira-Ticket</a> where I describe the problem and proposed a CSS-2.1 compatible definition:</p>
<pre>linear from( <size> , <size> ) to( <size> , <size> ) [ stop( <number> , <color> ) ]+ [ repeat | reflect ]?
radial [ center( <size> , <size> ) , ]? <size> [ focus( <size> , <size> ) ] ? [ stop( <number> , <color> ) ]+ [ repeat | reflect ]? /* Example */

</pre>

Usually, the semantic model is mapped to its graphical representation using a hard wired transformation, be it in code (<a rel="nofollow" target="_blank" href="http://www.eclipse.org/gef/">GEF</a>, <a rel="nofollow" target="_blank" href="http://www.eclipse.org/graphiti/">Graphiti</a>) or in models and a code generator (<a rel="nofollow" target="_blank" href="http://www.eclipse.org/modeling/gmp/">GMF</a>, the upcoming <a rel="nofollow" target="_blank" href="http://code.google.com/a/eclipselabs.org/p/spray/">Spray</a> project). Second, hard wiring the mapping introduces extra compilation/generation/deployment turnarounds.<br /><br />I've created a prototype of a graphical view that is configured using two textual DSLs: A mapping DSL that declaratively describes the transformation from semantic objects to graphical elements, and a styling DSL to modify the graphical representation. The graphical view interprets the mapping and the stylesheet on a given input model, giving immediate feedback on changes of these live models.<br /><br />Both DSLs are implemented in Xtext 2.0 and make heavy use of the new Xbase expression language library. As Xbase uses Java's typesystem, we can refer to any POJO as an input model. Xbase's rich semantics allow comfortably navigating the models references, and its interpreter can be customized easily to fit our needs. The graphics are implemented using plain GEF and <a rel="nofollow" target="_blank" href="http://www.eclipse.org/gef/zest/index.php">Zest</a> layouts.<br /><br />The following screencast gives you a short demo. APPlause is the missing link in my tool-chain of model-driven development of business applications running on desktops and also mobile.</p>
<p>Next week I’ll also talk at <a rel="nofollow" target="_blank" href="http://wiki.eclipse.org/Eclipse_DemoCamps_Indigo_2011/Hamburg">Democamp Hamburg</a>.</p>
